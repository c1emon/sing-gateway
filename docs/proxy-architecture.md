# 代理网关接入架构

## 1. 角色定义

本文不再使用 `W`、`C` 等字母代称，统一使用以下角色：

### 1.1 代理提供者 / 代理网关

**代理提供者 / 代理网关** 是运行 `sing-box` 的服务器，部署在主网关/防火墙可达的网络中，通常位于 DMZ 或服务网段。为提高可读性，后文统一简称为 **代理网关**。

代理网关可以提供以下能力：

- HTTP / SOCKS5 显式代理服务。
- FakeDNS / FakeIP DNS 服务。
- TPROXY 透明代理服务。
- 根据 FakeIP 映射还原域名。
- 根据规则执行代理出站、直连出站或拒绝访问。

在本文讨论的主要方案中，代理网关与被代理主机**不在同一二层网络**，不会直接响应被代理主机所在子网的 ARP。

### 1.2 被代理主机

**被代理主机** 是需要使用代理服务的客户端主机，可以通过以下方式接入代理：

- 应用或系统显式配置 HTTP / SOCKS5 代理。
- 使用代理网关提供的 FakeDNS 服务。
- 将默认网关配置为主网关/防火墙持有的 Proxy Gateway VIP，由主网关/防火墙策略路由到代理网关。

### 1.3 主网关 / 防火墙

**主网关 / 防火墙** 是连接被代理主机所在子网、DMZ、互联网和其他内部网络的核心网络设备。

本文中的主网关 / 防火墙不是特指 OPNsense。OPNsense 只是可选实现之一。该设备应抽象为具备以下能力的边界/核心网关设备：

- 在被代理主机所在子网持有 VIP / secondary IP，并响应 ARP/ND。
- 支持状态防火墙。
- 支持基于源地址、目的地址、协议、接口等条件的策略路由。
- 支持有状态 DNAT / port-forward，必要时支持受控 1:1 NAT。
- 支持出站 NAT。
- 支持规则优先级、健康检查和故障回退。

### 1.4 Proxy Gateway VIP

**Proxy Gateway VIP** 是主网关/防火墙在被代理主机所在子网中持有的虚拟 IP。

需要明确：

- Proxy Gateway VIP 由主网关/防火墙持有。
- 被代理主机 ARP 查询 Proxy Gateway VIP 时，由主网关/防火墙响应。
- 代理网关不持有该 VIP。
- 被代理主机将默认网关设置为 Proxy Gateway VIP 时，流量仍然首先进入主网关/防火墙，而不是直接进入代理网关。
- 被代理主机经该 VIP 转发普通流量时，IP 包的三层目的地址仍是最终目标地址，而不是 Proxy Gateway VIP。
- 主网关/防火墙通常不能仅凭“被代理主机使用了哪个默认网关 VIP”来匹配策略路由，除非具体设备支持基于入站 VIP、虚拟 MAC 或等价元数据匹配。
- 实际策略路由应优先基于源地址、入接口、VLAN、客户端组、协议、目的地址等条件匹配。

因此，本文讨论的透明代理网关模式不是“代理网关与被代理主机同二层并直接作为默认网关”，而是：

```text
被代理主机
  -> 主网关/防火墙持有的 Proxy Gateway VIP
  -> 主网关/防火墙策略路由
  -> 代理网关
  -> 主网关/防火墙
  -> 互联网或目标网络
```

## 2. 三种代理接入方式总览

| 方式 | 名称 | 被代理主机默认网关 | DNS 使用方式 | 主网关/防火墙职责 | 代理网关职责 | 覆盖范围 |
|---|---|---|---|---|---|---|
| 方式一 | 显式 HTTP/SOCKS5 代理 | 主网关/防火墙普通网关 | 任意 | 保证代理端口可达，可选 VIP DNAT | 提供 HTTP/SOCKS5 代理服务 | 配置了代理的应用/系统 |
| 方式二 | FakeDNS + FakeIP 路由 | 主网关/防火墙普通网关 | 直接使用代理 DNS，或由主网关/防火墙按策略转发 DNS 到代理 DNS | 将 FakeIP 段路由或策略路由到代理网关 | 提供 sing-box FakeDNS，持有 FakeIP 与域名映射，并代理 FakeIP 流量 | 经过 FakeDNS 解析并访问 FakeIP 的连接 |
| 方式三 | Proxy Gateway VIP + PBR 透明代理 | 主网关/防火墙持有的 Proxy Gateway VIP | 可使用代理 DNS、主网关 DNS 或其他 DNS；FakeIP 可选 | 持有 Proxy Gateway VIP，并按策略将流量导向代理网关 | 作为 TPROXY 透明代理 next-hop | 策略匹配的 TCP/UDP 流量，可包含真实 IP 与 FakeIP |

三种方式按配置复杂度和流量覆盖范围递增排列，但并非严格依赖关系，可独立使用，也可组合使用：

1. 方式一最简单、风险最低，但需要被代理主机或应用显式配置代理。
2. 方式二通过 FakeDNS 将域名解析结果转换为 FakeIP，再由主网关/防火墙把 FakeIP 流量送到代理网关。
3. 方式三接近网关级透明代理，被代理主机将默认网关配置为主网关/防火墙持有的 Proxy Gateway VIP，由主网关/防火墙进行策略路由导流。

## 3. 方式一：显式 HTTP/SOCKS5 代理

### 3.1 工作方式

代理网关在某个端口提供 HTTP 或 SOCKS5 代理服务。被代理主机的应用、系统代理设置或代理客户端显式配置该代理地址。

典型路径：

```text
被代理主机应用
  -> HTTP/SOCKS5 代理地址
  -> 主网关/防火墙路由
  -> 代理网关代理端口
  -> 代理出站
  -> 主网关/防火墙出站 NAT
  -> 互联网
```

代理地址可以是：

- 代理网关的真实 IP。
- 主网关/防火墙上的服务 VIP，再通过 DNAT / port-forward 转发到代理网关。

### 3.2 可行性

该方式完全可行，是最简单、最稳妥的接入方式。

核心要求只有：

- 被代理主机能够路由到代理网关的代理端口。
- 或者被代理主机能够访问主网关/防火墙上的服务 VIP，并由主网关/防火墙将对应端口 DNAT 到代理网关。

代理网关不需要与被代理主机在同一网段。

### 3.3 优点

- 架构简单。
- 不需要 TPROXY。
- 不需要 FakeIP。
- 不需要改变被代理主机默认网关。
- 故障影响范围小，只有配置了该代理的应用受影响。
- 容易审计、排错和灰度启用。

### 3.4 限制

- 需要被代理主机或应用显式配置代理。
- 不走系统代理的应用无法覆盖。
- UDP 支持取决于被代理主机、SOCKS5 支持能力和代理实现。
- 对“全主机透明代理”需求覆盖不足。

### 3.5 推荐配置原则

如果通过主网关/防火墙的 VIP 暴露代理服务，建议使用精确 DNAT / port-forward：

```text
Service VIP:HTTP_PROXY_PORT  -> 代理网关:HTTP_PROXY_PORT
Service VIP:SOCKS5_PORT      -> 代理网关:SOCKS5_PORT
```

不建议使用全端口 1:1 NAT 暴露代理网关。

显式代理端口应设置安全边界：

- 限制允许访问的来源网段或客户端组。
- 需要时启用代理认证或访问控制列表。
- 开启访问日志，便于审计。
- 避免将显式代理端口暴露到不可信网络。
- 如果通过 VIP 暴露代理端口，主网关/防火墙上只开放必要端口。

## 4. 方式二：FakeDNS + FakeIP 路由

### 4.1 工作方式

代理网关运行 sing-box，并提供 FakeDNS 服务。被代理主机的 DNS 查询进入代理网关后，sing-box 返回 FakeIP，并在自身维护 FakeIP 与域名的映射关系。

被代理主机随后访问 FakeIP。主网关/防火墙必须将 FakeIP 段路由或策略路由到代理网关，使代理网关能够收到目标地址仍为 FakeIP 的连接。

需要注意：FakeIP 流量到达代理网关时，目的地址仍是 FakeIP，而不是代理网关本机 IP。Linux 内核不会自动把这类流量交给 sing-box 处理。因此，代理网关必须具备透明入站接收机制，例如：

- TPROXY + fwmark + local route。
- TUN 入站。
- REDIRECT 或等价 NAT 重定向机制。
- 其他能让代理程序接管目标地址非本机流量的透明代理机制。

本文默认以 TPROXY 作为主要透明入站方式。

典型路径：

```text
DNS 阶段：

被代理主机
  -> 代理网关 DNS
  -> sing-box 返回 FakeIP
  -> sing-box 持有 FakeIP <-> 域名映射

访问阶段：

被代理主机
  -> FakeIP
  -> 主网关/防火墙
  -> 主网关/防火墙根据 FakeIP CIDR 路由/PBR 到代理网关
  -> 代理网关通过 FakeIP 还原域名
  -> 代理出站
  -> 主网关/防火墙出站 NAT
  -> 互联网
```

### 4.2 DNS 进入代理网关的方式

有两种常见方式。

#### 4.2.1 被代理主机直接使用代理 DNS

被代理主机 DNS 服务器配置为代理网关的 DNS 地址。

此时要求：

- 被代理主机能路由到代理网关 DNS 地址。
- 如果使用主网关/防火墙 VIP 暴露 DNS，主网关/防火墙应仅将 DNS 端口 DNAT 到代理网关。

推荐：

```text
DNS VIP:53/tcp,udp -> 代理网关:53/tcp,udp
```

#### 4.2.2 主网关/防火墙按策略转发 DNS

被代理主机 DNS 仍配置为主网关/防火墙。主网关/防火墙的 DNS forwarder / resolver 根据来源 IP、域名、客户端组或策略，将特定 DNS 请求转发给代理网关。

此方式适合：

- 只让部分来源 IP 使用 FakeDNS。
- 只让部分域名进入 FakeDNS。
- 保持被代理主机 DNS 配置不变。

注意：

- 只有被转发到代理网关的 DNS 查询才会生成 FakeIP。
- 没有进入代理网关 DNS 的域名不会生成 FakeIP，也不会被方式二覆盖。

### 4.3 主网关/防火墙必须配置 FakeIP 路由

FakeDNS 只负责返回 FakeIP，不能自动让 FakeIP 流量到达代理网关。

因此，主网关/防火墙必须配置 FakeIP CIDR 到代理网关的路由或策略路由。

示例：

```text
目的地址 = 198.18.0.0/15
下一跳 = 代理网关的 DMZ IP
```

或更精细：

```text
源地址 = 指定被代理主机网段
目的地址 = 198.18.0.0/15
协议 = TCP/UDP
下一跳 = 代理网关的 DMZ IP
```

关键要求：

- 主网关/防火墙只能对 FakeIP 段做 route/PBR，不能 DNAT FakeIP。
- 代理网关收到的包必须保留原始目标地址 FakeIP。
- 代理网关才能通过自身 sing-box FakeIP 映射还原域名。
- 主网关/防火墙上应将 FakeIP CIDR 从 bogon/drop/保留地址拦截策略中例外放行到代理网关。
- FakeIP CIDR 不应泄漏到 WAN，也不应被当作普通公网目的地址做普通出站 NAT。

### 4.4 可行性

该方式可行，且适合“按域名驱动代理”的场景。

成立前提：

1. 生成 FakeIP 的 DNS 服务必须是代理网关上的 sing-box。
2. 代理网关自然持有 FakeIP 与域名映射。
3. 被代理主机后续访问的 FakeIP 必须被主网关/防火墙路由或策略路由到代理网关。
4. FakeIP 段不能被主网关/防火墙作为 bogon、保留地址或异常目的地址直接丢弃。
5. 代理网关默认路由应回到主网关/防火墙，保证回程路径对称。
6. 代理网关上的透明入站规则能够接收目标地址为 FakeIP 的 TCP/UDP 流量。
7. FakeIP 映射生命周期与被代理主机 DNS 缓存需要匹配，避免被代理主机缓存旧 FakeIP 但代理网关映射已过期。

FakeIP 生命周期需要特别注意：

- DNS TTL 过长可能导致被代理主机长期缓存旧 FakeIP。
- sing-box 重启后 FakeIP 映射可能丢失，被代理主机缓存的旧 FakeIP 可能暂时无法还原域名。
- 多台代理网关场景下，FakeIP 映射如果不共享，同一被代理主机的 DNS 查询和后续连接必须落到同一代理网关，或使用一致性调度/共享状态机制。

### 4.5 覆盖范围

方式二覆盖：

- 经过代理网关 FakeDNS 解析的域名。
- 被代理主机随后访问 FakeIP 的 TCP/UDP 连接。

方式二不覆盖：

- 没有经过代理 DNS 的域名。
- 被代理主机直接访问真实 IP 的连接。
- 被代理主机使用 DoH/DoT/DoQ 绕过普通 DNS 的场景。
- 被代理主机缓存了真实 IP 的连接。
- 硬编码 IP 的应用。

因此，方式二不是全流量透明代理，而是 DNS/FakeIP 驱动的透明代理。

### 4.6 与 Proxy Gateway VIP 的关系

方式二中，Proxy Gateway VIP 可以用于暴露 DNS 服务，但不是必须。

例如：

```text
被代理主机 DNS = DNS VIP
DNS VIP 由主网关/防火墙持有
主网关/防火墙 DNAT DNS VIP:53 -> 代理网关:53
```

需要强调：

- VIP 仍由主网关/防火墙持有。
- 被代理主机默认网关仍可以是主网关/防火墙普通网关地址。
- FakeIP 访问阶段必须依赖主网关/防火墙上的 FakeIP 路由/PBR。

## 5. 方式三：Proxy Gateway VIP + PBR 透明代理

### 5.1 工作方式

被代理主机将默认网关配置为主网关/防火墙持有的 Proxy Gateway VIP。

主网关/防火墙响应该 VIP 的 ARP/ND，并接收被代理主机的默认路由流量。随后，主网关/防火墙根据策略路由规则，将符合条件的流量转发给代理网关。

代理网关作为 TPROXY 透明代理 next-hop，接收由主网关/防火墙转发来的 TCP/UDP 流量，并根据目标地址、FakeIP 映射、SNI/sniff、规则集等决定代理、直连或拒绝。

代理网关的默认路由必须指向主网关/防火墙，或者通过等价路由策略确保回程流量回到主网关/防火墙，以保持路径对称、状态防火墙可跟踪、出站 NAT 可生效。

典型路径：

```text
被代理主机
  -> 默认网关 = 主网关/防火墙持有的 Proxy Gateway VIP
  -> 主网关/防火墙
  -> 主网关/防火墙 PBR 到代理网关
  -> 代理网关 TPROXY
  -> 代理出站或直连出站
  -> 主网关/防火墙
  -> 互联网或目标网络
```

### 5.2 必须澄清的网关语义

虽然被代理主机把默认网关配置为 Proxy Gateway VIP，但该 VIP 是主网关/防火墙持有的地址。

因此：

- 被代理主机的二层下一跳是主网关/防火墙 MAC 或 HA 虚拟 MAC。
- 被代理主机流量首先进入主网关/防火墙。
- 代理网关不是被代理主机的同二层默认网关。
- 代理网关是主网关/防火墙策略路由选中的 next-hop。

更准确的描述是：

```text
主网关/防火墙持有 Proxy Gateway VIP，代理网关作为透明代理 next-hop。
```

不应描述为：

```text
代理网关直接成为被代理主机的默认网关。
```

### 5.3 DNS 是否必须使用 FakeIP

方式三中，FakeIP 不是必需条件，但推荐作为域名规则的稳定基础。

#### 使用 FakeIP

如果被代理主机 DNS 查询进入代理网关，并由 sing-box 返回 FakeIP，则方式三可以稳定按域名处理流量。

优势：

- 代理网关天然持有 FakeIP 与域名映射。
- 被代理主机访问 FakeIP 时，代理网关可准确还原域名。
- 域名规则比单纯依赖 SNI/sniff 更稳定。

要求：

- DNS 查询必须进入代理网关。
- FakeIP 目的流量必须被主网关/防火墙 PBR 到代理网关。

#### 不使用 FakeIP

被代理主机也可以使用普通 DNS，访问真实 IP。主网关/防火墙仍可将真实 IP 目的流量 PBR 到代理网关，代理网关通过 TPROXY 接收连接。

这种方式可行，但规则精度较低：

- HTTPS 主要依赖 SNI/sniff。
- ECH、无 SNI、IP 直连、部分 QUIC/UDP 场景会影响域名识别。
- 无法像 FakeIP 一样稳定建立“域名 -> 连接”的映射。

### 5.4 策略路由范围

方式三不应将所有流量无条件导向代理网关。

推荐主网关/防火墙策略路由只匹配需要透明代理的流量，例如：

```text
入接口 = 被代理主机所在子网接口
源地址 = 被代理主机网段或被代理主机组
协议 = TCP/UDP
目的地址 = 公网地址 或 FakeIP 段
下一跳 = 代理网关 DMZ IP
```

策略路由匹配依据应以网络可观测条件为准，例如源地址、入接口、VLAN、客户端组、协议和目的地址。被代理主机把默认网关设置为 Proxy Gateway VIP 后，普通转发包的目的地址仍是最终目标 IP，主网关/防火墙不一定能在 PBR 阶段直接按“使用了哪个 VIP 作为默认网关”匹配，除非设备明确支持此类元数据匹配。

推荐默认只对 TCP/UDP 做 PBR。ICMP 不应进入 TPROXY，尤其需要保持 PMTUD 相关 ICMP 可达，避免 MTU 黑洞。

必须排除：

- 主网关/防火墙自身接口地址和 VIP。
- 代理网关自身 IP。
- DMZ 网段。
- 所有内部网段。
- RFC1918、CGNAT、link-local、loopback、multicast、broadcast 等特殊地址。
- DHCP、NTP、内部 DNS、AD、LDAP、SMB 等基础设施服务。
- ICMP，或至少确保 PMTUD 相关 ICMP 不被破坏。
- VPN、IPsec、GRE、ESP 等特殊协议。
- 管理网段和故障恢复通道。

### 5.5 访问 Proxy Gateway VIP 上的服务

如果 Proxy Gateway VIP 同时作为被代理主机默认网关和代理服务/DNS 服务入口，则主网关/防火墙需要区分两类流量：

1. **访问 VIP 自身服务的流量**，例如 DNS 或显式代理端口。
2. **经由 VIP 作为默认网关转发的普通出站流量**。

推荐使用精确 DNAT / port-forward 处理 VIP 上的服务端口：

```text
Proxy Gateway VIP:53/tcp,udp      -> 代理网关:53/tcp,udp
Proxy Gateway VIP:HTTP_PROXY_PORT -> 代理网关:HTTP_PROXY_PORT
Proxy Gateway VIP:SOCKS5_PORT     -> 代理网关:SOCKS5_PORT
```

不建议默认使用：

```text
Proxy Gateway VIP:any -> 代理网关:any
```

原因：

- 暴露面过大。
- 容易误导运维人员认为 VIP 上所有端口都应转发到代理网关。
- 不利于区分“VIP 服务访问”和“默认网关转发”。
- 在不同网关设备上，1:1 NAT/binat/DNAT 的语义和执行顺序可能不同。

### 5.6 DNAT、PBR 与 SNAT 的顺序约束

Proxy Gateway VIP 同时承担“默认网关入口”和“服务入口”时，必须区分两类处理路径。

#### VIP 服务流量

访问 VIP 自身服务端口的流量应走 DNAT / port-forward：

```text
被代理主机 -> Proxy Gateway VIP:53
主网关/防火墙 DNAT -> 代理网关:53
代理网关 DNS 服务响应
主网关/防火墙执行有状态反向转换
被代理主机收到来自 Proxy Gateway VIP 的响应
```

#### 普通转发流量

被代理主机访问真实目标 IP 或 FakeIP 时，流量不应 DNAT 到代理网关，而应由主网关/防火墙通过 PBR 转发到代理网关：

```text
被代理主机 -> 真实目标 IP / FakeIP
主网关/防火墙 PBR -> 代理网关
代理网关透明入站接收并处理
```

落地前必须验证具体网关设备的 NAT、filter、route-to/PBR、状态表执行顺序。不同设备和不同规则类型可能存在差异。

主网关/防火墙在“被代理主机 -> 代理网关”的导流链路上不应做 SNAT，否则会破坏：

- 代理网关侧的真实源 IP 审计。
- 按源地址分流的规则。
- 透明代理对原始客户端身份的识别。

SNAT 应主要发生在代理网关代理出站之后，由主网关/防火墙对代理网关出网流量做出站 NAT；如果允许内核三层 bypass，则需单独设计对应 NAT 规则。

## 6. 三种方式的对比与适用场景

| 对比项 | 方式一：显式代理 | 方式二：FakeDNS + FakeIP 路由 | 方式三：Proxy Gateway VIP + PBR 透明代理 |
|---|---|---|---|
| 是否需要修改默认网关 | 否 | 否 | 是，改为主网关/防火墙持有的 Proxy Gateway VIP |
| 是否需要修改 DNS | 否 | 是，或由主网关/防火墙转发 DNS | 可选 |
| 是否需要 TPROXY | 否 | 需要透明入站能力，推荐 TPROXY，也可用 TUN/等价机制 | 需要透明入站能力，推荐 TPROXY |
| 是否需要 FakeIP | 否 | 是 | 可选，但推荐用于域名规则 |
| 是否覆盖硬编码 IP | 否 | 否 | 可覆盖，取决于 PBR 策略 |
| 是否覆盖未配置代理的应用 | 否 | 仅覆盖使用 FakeDNS 的连接 | 可以覆盖策略匹配流量 |
| 是否保留被代理主机源 IP 到代理网关 | 是 | 是 | 是 |
| 故障影响范围 | 小 | 中 | 大 |
| 配置复杂度 | 低 | 中 | 高 |
| 推荐场景 | 单应用、低风险、灰度接入 | 按域名透明代理 | 网关级透明代理 |

## 7. 必须补充到整体架构文档中的结论

### 7.1 三种方式共同依赖主网关/防火墙能力

主网关/防火墙应被抽象为能力模型，而不是绑定到某个具体产品。

它至少需要支持：

- 可被被代理主机访问的路由路径。
- 可选的 VIP / secondary IP。
- 有状态 DNAT / port-forward。
- 策略路由。
- 状态防火墙。
- 出站 NAT。
- 对 FakeIP 段和特殊地址段的可控处理。

### 7.2 VIP 所有权必须明确

Proxy Gateway VIP 明确由主网关/防火墙持有。

被代理主机将默认网关设置为 Proxy Gateway VIP 时：

```text
被代理主机仍然先进入主网关/防火墙。
```

不是：

```text
被代理主机直接进入代理网关。
```

### 7.3 FakeDNS 必须配套 FakeIP 路由

方式二和方式三中，只配置 FakeDNS 不够。

必须保证：

```text
FakeIP CIDR -> 代理网关
```

且应通过 route/PBR 保留目标地址 FakeIP，而不是 DNAT FakeIP。

### 7.4 TPROXY 排除规则必须与网关侧 PBR 配套

代理网关上的 TPROXY 规则不能无条件劫持所有 TCP/UDP。

需要排除：

- 本机服务地址与端口。
- DNS 监听端口。
- 显式代理端口。
- 管理端口。
- 内部网段和保留地址。
- 广播、组播、link-local。
- 不需要代理的基础设施服务。

网关侧 PBR 也必须做相同方向的约束，避免不应进入代理网关的流量被导入。

### 7.5 direct / bypass 语义必须明确

代理网关中所谓 `direct` 或 `bypass` 至少有两种含义：

1. **代理程序 direct 出站**：连接已经被 sing-box 接管，由 sing-box 自己直接连接目标。
2. **内核三层转发 bypass**：流量没有进入 TPROXY，Linux 内核直接将原始包转发回主网关/防火墙。

两者要求不同。

如果允许内核三层转发 bypass，则必须额外确认：

- 代理网关启用 IP forwarding。
- 代理网关默认路由指向主网关/防火墙。
- 主网关/防火墙允许从 DMZ 进入但源地址为被代理主机网段的流量。
- 主网关/防火墙出站 NAT 覆盖这类流量。
- 没有形成路由环路。

尤其要注意：如果主网关/防火墙对“源为被代理主机网段、目的为公网、协议为 TCP/UDP”的流量无条件 PBR 到代理网关，那么内核三层 bypass 从代理网关回到主网关/防火墙后，可能再次命中同一 PBR 规则并被送回代理网关，形成环路。

避免环路通常需要至少一种约束：

- PBR 仅匹配被代理主机所在 LAN 入接口，不匹配 DMZ 入接口。
- 对来自代理网关 DMZ 接口的 bypass 流量设置单独允许规则，不再 PBR 回代理网关。
- 使用 fwmark、路由表、状态匹配或策略标记区分已处理流量。
- 对 kernel bypass 流量明确是否允许保留源 IP，或是否需要在代理网关/主网关侧做受控 SNAT。
- 关闭或调整 DMZ anti-spoofing 检查，使其不会误拦截“入接口为 DMZ、源地址为被代理主机网段”的合法 bypass 流量。

## 8. 实现约束补充

### 8.1 DNS 策略

代理架构中 DNS 策略必须单独设计，不能只依赖默认 DNS 行为。

可选路径包括：

1. **普通 DNS**：被代理主机继续使用原 DNS。此时方式三仍可透明代理真实 IP 目的流量，但域名识别主要依赖 SNI/sniff，精度较低。
2. **DNS VIP DNAT**：被代理主机 DNS 指向主网关/防火墙持有的 DNS VIP，主网关/防火墙仅将 `53/tcp,udp` DNAT 到代理网关 DNS 服务。
3. **主网关/防火墙 DNS 转发**：被代理主机 DNS 指向主网关/防火墙，由主网关/防火墙根据来源 IP、域名、客户端组等策略将部分查询转发到代理网关 FakeDNS。
4. **DNS 劫持**：主网关/防火墙或代理网关拦截被代理主机发往外部的 DNS。该方式侵入性更强，必须排除内部 DNS、AD、DHCP、管理网段和代理网关自身 DNS 监听端口。

DoH、DoT、DoQ 需要单独策略处理：

- 如果允许它们直连，被代理主机可能绕过 FakeDNS，方式二将无法覆盖对应连接。
- 如果需要强制 FakeDNS，应在主网关/防火墙侧阻断或代理常见 DoH/DoT/DoQ 目的，或通过终端策略禁用。
- 如果通过透明代理处理 DoH/DoT/DoQ，需要确认不会破坏内部解析和认证业务。

FakeDNS 与普通 DNS 混用时，应明确哪些域名进入 FakeDNS、哪些域名走内部 DNS 或普通上游 DNS。内部域名、AD/LDAP/SMB、打印、监控、NTP 等基础设施域名通常不应进入 FakeIP。

### 8.2 TPROXY 系统前提

代理网关采用 TPROXY 时，至少需要满足：

- 使用 fwmark 标记需要交给本地代理程序处理的流量。
- 配置独立路由表，将对应 fwmark 的目标路由到本地 loopback，例如 `local 0.0.0.0/0 dev lo table <table>`。
- 关闭或放宽 `rp_filter`，避免来自其他子网的源地址在 DMZ 入接口被 Linux 丢弃。
- 对本机出站流量设置防环路机制，例如 ignore mark、ignore UID、cgroup、专用出站 mark 或单独路由表。
- TPROXY 规则应限定入接口，避免误劫持 Docker、WireGuard、管理接口或其他虚拟接口流量。
- TPROXY 监听端口不应作为普通服务端口暴露给被代理主机直接访问。

### 8.3 IPv6

IPv6 必须明确选择策略，不能默认继承 IPv4 设计。

可选策略：

1. **暂不支持 IPv6**：在被代理主机侧、主网关/防火墙侧或代理网关侧明确禁用、绕过或阻断 IPv6，避免 IPv6 绕过 IPv4 代理策略。
2. **IPv6 直连绕过**：明确 IPv6 不进入代理，仅 IPv4 进入代理。此时需要接受 IPv6 可能绕过代理策略的事实。
3. **完整支持 IPv6**：需要额外设计 NDP/RA、IPv6 PBR、FakeIPv6、TPROXY v6、IPv6 出站策略，以及 NAT66/NPTv6 或无 NAT 出口模式。

如果启用 IPv6 FakeIP，需要确认：

- FakeIPv6 段不会被主网关/防火墙丢弃。
- IPv6 PBR 能将 FakeIPv6 流量送到代理网关。
- 代理网关具备 IPv6 TPROXY/TUN 接收能力。
- ICMPv6，尤其是邻居发现和 PMTUD，不被错误导入 TPROXY 或阻断。

### 8.4 MTU、MSS 与 ICMP/PMTUD

代理链路、隧道出站、VPN 出站或多次转发可能降低有效 MTU。

建议：

- PBR 默认只导流 TCP/UDP，不导流 ICMP。
- PMTUD 相关 ICMP/ICMPv6 必须允许回到被代理主机。
- 对 TCP 可在主网关/防火墙或代理网关侧设置 MSS clamping。
- 对 UDP/QUIC 应关注分片、路径 MTU 和代理软件的 UDP fragment 处理能力。

### 8.5 HA、健康检查与故障回退

代理网关故障策略必须提前定义。

可选模式：

- **Fail-closed**：代理网关不可用时，被代理主机出站失败。安全性较高，但可用性较差。
- **Fail-open**：代理网关不可用时，主网关/防火墙撤销 PBR，让流量走普通出口。可用性较高，但可能绕过代理策略。
- **多代理网关冗余**：主网关/防火墙根据健康检查将 PBR 切换到其他代理网关。

多代理网关与 FakeIP 组合时要特别注意：

- DNS 查询和后续 FakeIP 连接应落到同一代理网关；或
- 多代理网关共享 FakeIP 映射状态；或
- 使用一致性调度确保同一被代理主机稳定命中同一代理网关。

### 8.6 安全暴露面

需要最小化代理网关暴露面：

- 显式代理端口应限制来源并启用 ACL/认证。
- DNS 服务只允许被代理主机网段或主网关/防火墙访问。
- TPROXY 端口不应作为普通服务端口开放。
- 管理端口应仅允许管理网段访问。
- 禁止将全端口 1:1 NAT 作为默认暴露方案。
- 代理网关本机防火墙应只开放 DNS、显式代理、健康检查、管理等必要端口。

### 8.7 与原始架构文档的关系

本文用于定义角色、接入模式和网络边界。以 OPNsense 为例的具体配置步骤见 [`opnsense-proxy-gateway.md`](opnsense-proxy-gateway.md)，并应建立在本文能力模型之上，明确哪些配置是 OPNsense 实现细节，哪些是通用网关设备要求。

## 9. 推荐最终表述

建议将整体方案表述为：

> 本架构包含被代理主机、主网关/防火墙、代理网关三个角色。代理网关运行 sing-box，提供显式代理、FakeDNS/FakeIP 和 TPROXY 透明代理能力。主网关/防火墙持有被代理主机所在子网的 Proxy Gateway VIP，并负责 VIP ARP/ND、DNAT/port-forward、策略路由、状态防火墙和出站 NAT。被代理主机可以通过三种方式接入代理：显式 HTTP/SOCKS5 代理、FakeDNS + FakeIP 路由、Proxy Gateway VIP + PBR 透明代理。Proxy Gateway VIP 始终由主网关/防火墙持有，代理网关在透明代理模式中是主网关/防火墙策略路由选中的 next-hop，而不是与被代理主机同二层的真实默认网关。

## 10. 最终结论

三种方式覆盖了代理网关为被代理主机提供代理服务的主要接入模型，且在明确前提条件后均可行。

### 10.1 方式一结论

显式 HTTP/SOCKS5 代理方式最简单、最稳定，适合作为基础能力和灰度接入方式。它只要求被代理主机能够路由访问代理网关的代理端口，或通过主网关/防火墙 VIP DNAT 到代理端口。该方式不提供全透明能力。

### 10.2 方式二结论

FakeDNS + FakeIP 路由方式可行，前提是 DNS 查询进入代理网关上的 sing-box，并且主网关/防火墙将 FakeIP 段路由或策略路由到代理网关。该方式适合按域名透明代理，但只能覆盖使用 FakeDNS 并访问 FakeIP 的连接。

### 10.3 方式三结论

Proxy Gateway VIP + PBR 透明代理方式可行，是最接近“网关级透明代理”的模式。被代理主机默认网关配置为主网关/防火墙持有的 Proxy Gateway VIP，流量先进入主网关/防火墙，再由主网关/防火墙策略路由到代理网关。该方式可以配合 FakeIP，也可以处理真实 IP 目的流量，但必须严格设计 PBR 范围、TPROXY 排除规则、DNS 策略、故障回退和 bypass 语义。

### 10.4 落地必须满足的架构约束

1. 代理网关与被代理主机不在同一二层时，不应描述为代理网关直接成为被代理主机网关。
2. Proxy Gateway VIP 必须明确由主网关/防火墙持有。
3. FakeDNS 必须配套 FakeIP 路由或策略路由。
4. VIP 上服务转发推荐使用精确 DNAT / port-forward，不推荐全端口 1:1 NAT。
5. 主网关/防火墙 PBR 不能简单配置为 `source = 被代理主机网段, destination = any, gateway = 代理网关`。
6. 代理网关 TPROXY 不能无条件劫持所有 TCP/UDP。
7. direct/bypass 必须区分代理程序 direct 出站与内核三层转发 bypass。
8. IPv6、MTU/MSS、ICMP/PMTUD、HA、健康检查、故障回退和安全暴露面必须按第 8 节明确策略后再落地。
